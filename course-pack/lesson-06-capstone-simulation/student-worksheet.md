# Student Worksheet: Lesson 6

## Theory (What You Need to Know)
- Plan first, then code: this prevents random errors.
- A test table predicts outputs before you run the program.
- Good design means clear variable names and fair rule thresholds.

## Metalanguage (Key Terms + Examples)
- `algorithm`: Step-by-step method to solve a problem.
Example: input data -> calculate score -> choose outcome -> print report.
- `pseudocode`: Plain-language version of code logic before coding.
Example: "If score >= 50, set outcome to High".
- `logic map`: Ordered decision plan for branches.
Example: first check high score, then medium, else low.
- `test table`: Planned input/output checks written before running code.
Example: Test A expected "Approved", then compare with actual result.
- `validation`: Checking if input is sensible for your rules.
Example: if expected range is 0-100, negative values are invalid.
- `readability`: How easy code is to understand.
Example: `risk_score` is clearer than `x`.
- `evidence`: Proof that your code works from test outputs.
Example: three test runs where expected equals actual.

## Relatable Examples
- Game designers test easy, medium, and hard runs before release.
- Film scenes are storyboarded before filming.
- Competition rules should be easy to explain and defend.

You are successful when your project:
1. Meets all 4 required features
2. Includes a complete 3-case test table (expected vs actual)
3. Produces valid output for each test
4. Includes one reflection on a bug and one design decision
5. Runs without forbidden topics

## Scaffolded Task (Do in Order)
1. Choose one scenario.
- Arena checkpoint, maze terminal, house ranking, or portal gate.
2. Plan your variables.
- Write names and purpose for at least five variables before coding.
3. Draft your logic map.
- Write the three branch conditions on paper first.
4. Write input section.
- Add at least three `input()` lines and convert each to numbers if needed.
5. Add data structure.
- Create one meaningful list or tuple used in your final decision/output.
6. Write branch logic.
- Implement `if/elif/else` with at least three outcomes.
7. Print final report.
- Print key inputs, computed value, and final decision.
8. Validate with your test table.
- Run Test A, Test B, Test C and compare expected vs actual outputs.
